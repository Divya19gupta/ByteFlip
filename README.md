# ByteFlip

A personal DSA revision tool. Track which problems you've solved, then revise them as flashcards in a shuffled queue, with AI-generated hints when you're stuck.

**Live app:** _[ByteFlip](https://byte-flip.vercel.app/)_

## Features

- **305 DSA problems**: across 13 topics (arrays, graphs, DP, trees, and more), each with brute-force, optimal approach, time/space complexity, and a key trick.
- **Track solved problems**: tick problems off as you solve them, organized by category, with an at-a-glance progress meter.
- **Revision sessions**: solved problems enter a shuffled queue; work through them one at a time, reveal answers progressively (brute force → optimal → complexity → trick), and rate your confidence. Sessions resume where you left off, and a round finishes with a "Finish" button back to the problem list.
- **AI hints**: stuck on a problem? Get a short, non-spoiler hint generated by Gemini, grounded in semantically similar problems found via a vector search over the problem set.

## Tech stack

- **Backend:** FastAPI (Python)
- **Frontend:** React + Vite
- **Vector search:** ChromaDB, using Gemini's embedding API (`gemini-embedding-001`) rather than Chroma's default local model, keeps the backend lightweight enough to run on a free hosting tier
- **AI hints:** Gemini API (`gemini-2.5-flash`)
- **Storage:** flat JSON files (`problems.json`, `solved.json`, `session.json`) [no database]
- **Hosting:** backend on Render, frontend on Vercel

## Project structure

```
ByteFlip/
├── backend/
│   ├── main.py            # FastAPI app: all API endpoints
│   ├── embeddings.py      # one-time script: embeds all problems into ChromaDB via Gemini
│   ├── requirements.txt
│   ├── problems.json      # the 305-problem dataset
│   ├── solved.json        # ids of solved problems (gitignored, created automatically)
│   ├── session.json       # revision queue/seen state (gitignored, created automatically)
│   └── chroma_db/         # vector store: committed to the repo (see note below)
└── frontend/
    └── src/
        ├── pages/          # HomePage, ProblemsPage, RevisionPage
        └── components/     # Navbar, PageHeader, ByteMeter
```

**Why `chroma_db/` is committed, unlike the other generated files:** `problems.json` never changes, so its embeddings never need to change either. Building them requires ~300 calls to Gemini's embedding API, which is slow and rate-limited on the free tier, fine to do once, on your own machine, but not something that should run automatically every time the backend deploys. Committing the finished result means the live server just loads it instead of rebuilding it. `solved.json` and `session.json`, by contrast, are per-user runtime state, so they're gitignored and recreated automatically on first run.

## Setup

### Backend

```bash
cd backend
python -m venv ../venv
source ../venv/bin/activate      # Windows: ..\venv\Scripts\activate
pip install -r requirements.txt
```

Create a `.env` file in `backend/`:

```
GEMINI_API_KEY=your_key_here
```

If `chroma_db/` isn't already present (e.g. you edited `problems.json`), build it once:

```bash
python embeddings.py
```

This takes a few minutes, it embeds problems in small batches with pauses between them to stay under Gemini's free-tier rate limit.

Run the server:

```bash
uvicorn main:app --reload
```

Backend runs at `http://localhost:8000`. On first run, `solved.json` and `session.json` are created automatically if missing.

### Frontend

```bash
cd frontend
npm install
```

Create a `.env` file in `frontend/`:

```
VITE_API_URL=http://localhost:8000
```

```bash
npm run dev
```

Frontend runs at `http://localhost:5173`.

## Environment variables

| Variable | Where | Purpose |
|---|---|---|
| `GEMINI_API_KEY` | `backend/.env` | Used for both embedding problems (via `embeddings.py`) and generating hints (via `/hint`) |
| `VITE_API_URL` | `frontend/.env` | Backend URL: `http://localhost:8000` locally, your Render URL in production |

## Deployment

**Backend (Render):**
- Root directory: `backend`
- Build command: `pip install -r requirements.txt`
- Start command: `uvicorn main:app --host 0.0.0.0 --port $PORT`
- Environment variable: `GEMINI_API_KEY`

**Frontend (Vercel):**
- Root directory: `frontend`
- Framework preset: Vite (auto-detected)
- Environment variable: `VITE_API_URL` → your deployed Render backend URL

## Known limitations

- Storage is flat JSON files, not a database, fine for local/personal use, not safe for multi-user or concurrent access.
- On Render's free tier, `solved.json` and `session.json` reset on redeploy (the disk is ephemeral), they're recreated empty automatically, but existing progress is lost. `chroma_db/` is unaffected since it's committed to the repo, not generated at runtime.
- Render's free tier spins the backend down after inactivity; the first request after idling can take up to ~30 seconds while it wakes back up.
- No authentication: this app assumes a single user.
